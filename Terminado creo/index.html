S W E B O K      v  3.0

SWEBOK, Software Engineering Body of Knowledge, es un documento creado por la Software Engineering Coordinating Committee, promovido por el IEEE Computer Society, que se define como una guía al conocimiento presente en el área de la Ingeniería del Software.

Requerimientos


Esta área se enfoca en los principios y prácticas relacionadas con la gestión de los requisitos del software, fundamentales para el éxito de cualquier proyecto de software.

Requisitos funcionales

Los requisitos funcionales describen lo que el sistema debe hacer, detallando las funciones y comportamientos específicos que debe proporcionar para cumplir con los objetivos planteados. Incluyen tareas como procesar datos, generar reportes o interactuar con el usuario.

Ejemplos
  • El sistema debe permitir a los usuarios registrarse con un nombre y correo electrónico.
  • El sistema debe enviar notificaciones automáticas al completar una compra.
  • El usuario debe poder filtrar productos por categoría y precio.

Requisitos no funcionales

Los requisitos no funcionales describen las características de calidad que el sistema debe cumplir, como rendimiento, usabilidad, seguridad o disponibilidad. Estos requisitos se enfocan en cómo debe comportarse el sistema más allá de sus funciones principales.

Ejemplos
  • Las actualizaciones de seguridad deben ser aplicadas automáticamente en un plazo de 24 horas desde su disponibilidad.
  • El sistema debe ser compatible con los navegadores más recientes, incluyendo Chrome, Firefox y Safari.
  • El sistema debe realizar copias de seguridad automáticas cada 24 horas y almacenarlas en un entorno seguro.

ELICITACIÓN:

Recopilar y descubrir los requisitos del sistema.

ANÁLISIS:

Refinar y clasificar los requisitos

ESPECIFICACIÓN:

Documentar los requisitos para los interesados.

VALIDACIÓN:

Asegurar que los requisitos son correctos y completos.

IMPORTANCIA:

Asegura que el software cumpla con las expectativas de los usuarios y los objetivos del proyecto.

Las pruebas son una parte crucial del proceso de desarrollo de software. Consisten en verificar y validar que el sistema funcione correctamente y cumpla con los requisitos especificados. Se realizan en diferentes etapas del ciclo de vida del software para identificar defectos, mejorar la calidad del producto y garantizar que se cumplan las expectativas del usuario final.










Mantenimiento

El Mantenimiento de software en SWEBOK se describe como el proceso de realizar cambios al software después de su implementación inicial. Las categorías de mantenimiento, como correctivo, adaptativo, perfectivo y preventivo, se abordan en SWEBOK con definiciones y explicaciones detalladas sobre cada tipo:

Correctivo

Soluciona errores y defectos identificados después de la implementación.

Adaptativo

Permite que el software siga siendo útil al ajustarlo a nuevos requisitos del entorno, como cambios en hardware, plataformas o sistemas operativos.

Perfectivo

Mejora las características del software para hacerlo más eficiente, añadir funcionalidades o mejorar la experiencia del usuario.

Preventivo

Realiza modificaciones para reducir la posibilidad de problemas futuros, como mejorar la mantenibilidad o la estabilidad.















Gestión de configuración

La GCS es un proceso sistemático para identificar, organizar y controlar las versiones y configuraciones del software y sus componentes a lo largo de su ciclo de vida. Su objetivo es asegurar que el software se desarrolle y mantenga de manera ordenada, permitiendo la trazabilidad de cambios y la estabilidad del producto.






Gestión de Identificación de Configuración

Consiste en identificar de forma única todos los elementos de configuración de un sistema (código fuente, documentación, archivos de pruebas, etc.).

Control de Configuración

Establece y supervisa las políticas para gestionar los cambios en los elementos de configuración.

Gestión de Versiones

Se ocupa de registrar y rastrear múltiples versiones de un sistema de software.

Gestión de Cambios

Controla las modificaciones realizadas en los elementos de configuración, garantizando que todos los cambios sean planificados y documentados.

Auditoría de Configuración

Verifica que los elementos de configuración cumplen con los requisitos especificados y que las versiones liberadas están completas y correctas.


Tipos:

Auditoría Funcional: Verifica que el software cumple con las funcionalidades requeridas.

Auditoría Física: Confirma que todos los elementos están presentes y documentados correctamente.
